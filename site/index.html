How it works

You play inside Shoup's oracle

The group is a black box. You never see coordinates — only opaque, per-run-random 128-bit tokens — so the operation count the oracle keeps is unforgeable. There is no way to do a group operation off-meter.

1

The oracle holds the secret

A real curve E: y²=x³+ax+b over F_p with prime order n, a generator P, a secret scalar k, and the target Q = k·P. It hands you only tokens for P, Q, O.

2

A counted interface

add costs +1. neg is free (−P=(x,−y)). scalar_mul costs its doublings + adds. Batched add_batch / neg_batch step many walks in one round trip — every op still counted.

3

Return k, get scored

The oracle checks k·P = Q on the real curve and writes its operation counter to score.json. The score is the mean over several trials — rho's collision time has heavy variance.

oracle session — every add is metered
solver ▸ add(t_a, t_b)            → t_c          [ +1 op ]
solver ▸ neg(t_c)                 → t_-c         [ free  ]
solver ▸ add_batch(64 parallel walks) → 64 tokens    [ +64   ]
          … distinguished-point negation-map rho …
solver ▸ submit(k)
oracle ▸ k·P == Q  ✓   score = 708,536 group ops   (0.72× rho)
No loopholes. Tokens are an 8-round Feistel permutation of the point, re-keyed every run — you can only name elements you reached through counted queries. The counter lives in a separate trusted process; the solver is sandboxed with no network and no filesystem writes; the secret seed is fresh and never revealed. You cannot under-report, pre-seed, replay, or read the answer off disk.

The benchmark

Will you beat rho?

Pollard's rho is the best known general-purpose attack on the discrete-log problem: about √(πn/2) ≈ 1.2533·√n group operations. Shoup's generic lower bound says no black-box algorithm does asymptotically better than √n. This challenge turns that argument into a measurable game — a constant-factor arena. How close to the floor can a real implementation get?

Participate

Edit one file. Run. Score.

A Rust harness with two halves separated by a process boundary: the trusted oracle, and your solver. You edit exactly one file — src/solver/mod.rs — and the sandbox does the rest.

# clone & build the oracle, solver, and instance generator $ git clone https://github.com/aburan28/ecdlp-cost-challenge $ cd ecdlp-cost-challenge $ ./setup.sh # the ONE file you may change — your algorithm lives here $ $EDITOR src/solver/mod.rs # sandboxed run: builds, confines the solver, scores the mean $ ECDLP_BITS=40 ./benchmark.sh --note "tried Gaudry–Schost" $ cat score.json # your score; one row appended to results.tsv

The other half

First-Blood board — the representation track

The scored arena hides the curve, so it can only measure generic algorithms, where √n is a proven floor. This board is the opposite half: it publishes the full curve (p, a, b, G, Q) and dares you to recover k by any method — generic or not. k was generated at random and discarded; nobody knows it.

Scope & honesty

What the score does — and doesn't — mean

It's the GGM, executed

Hiding the representation is what makes the op-count unforgeable — but it also means the scored arena cannot host non-generic attacks. Inside it, the expected score cannot go below √(n/2); the only game is the constant factor.

"Generic" is enforced

Every instance has prime order, n ≠ p, large MOV embedding degree, and j ∉ {0,1728}. The generator computes the order exactly and Sage re-checks it — a win can never come from an accidentally weak curve.

Sizes are toy by design

Scored tiers are small enough to actually solve (bits ≤ 48). This benchmarks algorithmic group-operation efficiency — it is not a break of any deployed parameter set. Nothing here recovers a real key.

A research instrument

Not a claim that ECDLP is broken. It asks one clean, finite question: how close to the rho optimum — and to the Shoup floor — can a real implementation get?
